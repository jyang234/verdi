Phase-5 review disclosure (appended per ADJ-67/D6-38; append-only — the claim above stands unaltered). The adversarial verifier found the one input this family's dc-1 "closure-gate semantics byte-for-byte unchanged" certification missed. The round replaced the fold's stat-only AttestationExists with content-reading LoadAttestationState; on an attestation file that is present but unreadable (mode 000, foreign uid, or ACL), the two are NOT byte-identical — the old stat-only path swallowed the read to attested=true and the closure gate computed a verdict (exit 0/1), while the kept LoadAttestationState propagates the os.ReadFile error so both the story and feature closure gates fail operationally (exit 2). I reviewed this directly: the fail-closed behavior is correct and is KEPT — an unreadable file silently counting as a satisfied HUMAN attestation was the latent honesty defect the three-valued-honesty rule forbids, and an unreadable store is an operational error, not a verdict. So the feature's dc-1 byte-for-byte promise is violated-with-witness on this single IO-error input: the certification was wrong there, the behavior is right, and the record now says so. Filed D6-38; pinned by permission-denied characterization tests on both gates (skipped when euid==0).
