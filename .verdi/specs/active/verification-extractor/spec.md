# Verification Extractor

## Problem

A `class: proposal` diagram (`proposal-artifact`) can now carry `scope` and
`derived_from`, but nothing turns its mermaid body into a comparable
structure, nothing regenerates truth from the pinned flowmap CLI at the
proposal's scope, and nothing runs the three-way comparison `ac-1`
promises: exists / proposed-new / kept-but-gone-contradicted-with-witness /
stale-base. Without it every proposal is an unverified drawing regardless
of what it claims.

## Outcome

A declared, honestly-bounded mermaid-flowchart vocabulary is extracted
one-way into a comparable node/edge set. Truth is regenerated by execing
the pinned flowmap CLI at the proposal's scope and strict-decoding its
graph JSON. A three-way structural comparison discloses
exists/proposed-new/kept-but-gone (with the removing commit as witness) per
element, plus a separate stale-base check by digest comparison against
`derived_from`. Coverage (full vs. partial) is disclosed per artifact
rather than silently assumed. No LLM and no rename inference appear
anywhere in the computation.

## AC-1

A declared mermaid-flowchart grammar subset (DC-1) is parsed into a
node/edge set with an identity-normalization rule (DC-2). Any construct
outside the grammar downgrades the WHOLE artifact's coverage to `partial`
rather than silently mis-parsing or silently claiming full coverage. Full
coverage requires every declared node/edge to parse cleanly AND normalize
to an unambiguous truth-comparable identity. Evidence: static (the grammar
is named, the parser's recognized token forms enumerated in code, and the
coverage-classification rule stated) + behavioral (table-driven parser
tests: every recognized node/edge form, a diagram using an unrecognized
construct downgrading to partial, and an identity collision downgrading to
partial).

## AC-2

Truth is regenerated by execing the pinned flowmap CLI's `graph` subcommand
at the proposal's declared `scope` (or unscoped when absent) and
strict-decoding its JSON through the existing `internal/upstream` seam
(`RunGraph`/`DecodeGraph`). Hermetic tests consume canned captures
(`testdata/svcfix-canned`), never a live exec. Evidence: static (the exec
request shape and the decode call site) + behavioral (a test proving the
scoped and unscoped invocations both decode a canned graph correctly, and
an operational-error test for a non-zero flowmap exit).

## AC-3

A three-way structural comparison between the proposal's extracted set and
regenerated truth classifies every element as `exists` (proposal element
present in truth), `proposed-new` (proposal element absent from truth, no
base to have dropped it from), or `kept-but-gone`/contradicted (an element
the proposal inherited from its base that truth no longer has), the latter
carrying a candidate witness commit (DC-4) when one resolves — disclosed
as a candidate, never overclaimed as causally verified. A rename
renders as two independent facts — the old identity `kept-but-gone`, the
new identity `proposed-new` — never inferred as a single "renamed" fact
(parent DC-5: no rename inference, honest witnesses over concise
inference). Evidence: static (the comparison function's three-valued
output type, with no fourth "renamed" value anywhere in it) + behavioral
(a table-driven test over exists/proposed-new/kept-but-gone/rename-as-two-facts,
plus a witness-resolution test and a witness-absent test over a fixturegit
history). The comparison's result type (including its `exists`/
`proposed-new`/`kept-but-gone` classification and candidate-witness/
witness-unresolved labels) is a COMPUTED, unpersisted-to-artifact Go
result — rendered disclosure, never a frontmatter field the diagram
artifact strict-decodes — so it is not schema surface within parent DC-7's
ratified 02 amendment batch (which governs the diagram artifact's own
persisted fields: status, scope, derived_from) any more than
`internal/align`'s existing `Finding`/`ComputedResult` types are.

## AC-4

For a derived proposal, stale-base is detected by recomputing the base's
canonical graph-JSON digest at current HEAD (same scope, same flowmap
invocation AC-2 already performs) and comparing it against
`derived_from.digest` (DC-5). A mismatch discloses `stale-base`
independently of AC-3's own three-way result — a proposal can be
`stale-base` and still have every element `exists`, or vice versa; the two
checks are orthogonal and both run. Evidence: static (the digest
recomputation call site and the byte-comparison) + behavioral (a test with
a matching digest disclosing no staleness, and one with a deliberately
mismatched digest disclosing `stale-base`).

## DC-1

The claimed grammar: a direction line (`flowchart|graph <DIR>`, `DIR`
ignored — layout is renderer-owned, `co-3` of the parent feature), node
declarations of the bare-id, id+shape-delimited-quoted-label, or
id+shape-delimited-quoted-label`:::classname` forms (the trailing
`:::classname` node-class assignment flowmap's own renderer emits for
fallible/db/bus/external/blind nodes — presentation only, never part of
identity), `%%` comment lines (skipped — flowmap's own renderer always
emits at least one header comment, so excluding comments from the grammar
would make a PRISTINE, freshly generated diagram fail to reach full
coverage, contradicting the parent's `dc-3` "full = within the generator's
vocabulary"; a judged-sweep finding on this story's own design caught
exactly this gap, corrected here rather than dispositioned away), a
`classDef ...` declaration line (skipped, presentation only — the same
generator-fidelity reason), and edge declarations
`id --> id` / `id -->|label| id` / `id -.-> id` / `id -. label .-> id`
(arrow style and label text are read but NOT part of identity — only the
ordered `(from, to)` id pair is). Anything else — `subgraph` blocks (not
merely assumed absent: verdi-go's `graphio/mermaid.go` `Mermaid()`
renderer — the plain call-graph view this story's truth regeneration
targets — emits node declarations, boundary/disclosure nodes, edges, and
`classDef`s, but no `subgraph` anywhere in its source; `subgraph` lanes
appear only in the separate `rollup_mermaid.go`
`RollupMermaidOptions.Bands` view, a different C3-altitude renderer this
story does not target), `click` bindings, any other mermaid diagram type's
syntax — remains OUTSIDE the grammar. Coverage is an
artifact-wide, not a line-by-line, verdict (the parent feature's `dc-3` is
read as classifying the WHOLE flowchart): the instant one line falls
outside the declared grammar, the artifact's coverage is `partial`, though
the extractor still attempts a best-effort parse of the recognized lines
for disclosure — an element whose parse was uncertain is excluded from the
three-way comparison rather than guessed.

## DC-2

Identity normalization: a proposal's raw mermaid node-id token is compared
against `shortName(fqn)` for every truth node, where `shortName` is a
small function THIS package defines and tests (strip a leading
`(*pkg.Type).` or `pkg.Type.` receiver/package prefix, keep the trailing
name) — never verdi-go's own `frontier.ShortName` (CLAUDE.md: never import
verdi-go packages). Two truth nodes whose `shortName` collides make that
name AMBIGUOUS: a proposal node using it cannot be classified with
confidence and the WHOLE artifact falls to `partial` coverage (DC-1) rather
than guessing which truth node was meant. This is a display-name
convenience, not a reimplementation of flowmap's own call-graph
construction (the parent feature's `co-2` forbidden reimplementation), and
it is honestly narrower than flowmap's internal collision-disambiguation
scheme — disclosed, not hidden.

## DC-3

Truth regeneration execs `flowmap graph -entry <scope> <dir>` when `scope`
is present, or `flowmap graph <dir>` unscoped, through `internal/upstream`'s
existing `RunGraph`/`DecodeGraph` strict-JSON seam (extended with an
optional entry-scope argument; no new exec surface). The parent feature's
`dc-6` names the proposal's `scope` field after flowmap's `--root`
selector, but `--root` is documented (verdi-go's `cmd/flowmap/main.go`) as
a MERMAID-render-time-only scoping flag, not a JSON-mode one; `--entry` is
the JSON-decodable build-time equivalent over the same selector value
space — not merely asserted: verdi-go's `cmd/flowmap/main.go` own
`--help` text for BOTH flags uses the identical example
`"POST /loan-application"`, concrete textual evidence the two flags share
one selector grammar. This is the smallest reversible reading of `dc-6`
that stays inside `co-2`'s JSON-only truth channel, disclosed as a
build-time-verifiable reading rather than a closed fact: if a real flowmap
release ever accepts a `--root` value `--entry` rejects (or vice versa),
that is a build-time finding for this story's implementer to record in
`PLAN.md`'s invention ledger, not a silent assumption here. A future
flowmap release exposing root-scoped JSON directly is a one-line swap
behind this same seam. Unscoped regeneration accepts `dc-6`'s disclosed
hairball-cap case as-is — capping/indexing presentation is a different
story's concern.

## DC-4

Witness-commit discovery: `git log -S<identity-string> -1 --format=%H --
<service-dir>` (fixed-string pickaxe, most recent hit) over the element's
normalized identity string — the smallest honest mechanism available
without building history-walking graph-diff machinery. The result is
disclosed as a CANDIDATE witness commit, never a causally-verified one: a
fixed-string pickaxe hit only proves the identity string's occurrence
count changed in that commit somewhere under the service directory (a
comment, a test, an unrelated symbol sharing the name) — it does not prove
that commit removed THIS graph element. This candor is itself a
correction: an earlier draft's "the removing commit" phrasing overclaimed
a certainty this mechanism cannot deliver, a gap this story's own judged
sweep caught and this decision now names explicitly rather than disclosing
only the zero-hit case below. When the search finds no hit at all (a moved
directory, an unrelated rename) the witness is disclosed as unresolved
rather than fabricated or blocked — `kept-but-gone` still renders either
way, labeled candidate-witness or witness-unresolved, honestly short of
causal proof rather than silently overclaiming it.

## DC-5

Stale-base recomputes the base's canonical graph JSON at current HEAD
using the SAME flowmap invocation AC-2/DC-3 already performs (same scope),
canonicalizes it via the shared 02 §Generated artifacts and digests formula
(the same `canonjson.Digest` this codebase already uses for every other
computed digest), and compares the result to `derived_from.digest`
byte-for-byte. This story detects and discloses the mismatch only; the
rebase affordance the mismatch motivates (the parent feature's `ac-3`,
"disclosed with a rebase affordance") is board-editor's concern, out of
scope here.

## CO-1

No LLM anywhere in this story's code (parent `co-1`): grammar parsing,
identity normalization, truth regeneration, the three-way comparison,
witness discovery, and the digest comparison are all pure/deterministic or
hermetically-exec'd Go.

## CO-2

No network in any test (parent `co-2`): flowmap exec is tested through
`internal/upstream`'s existing fake-`Runner` seam over
`testdata/svcfix-canned` captures, never a live binary; witness-commit
discovery is tested over a fixturegit repository with a scripted, known
history.
