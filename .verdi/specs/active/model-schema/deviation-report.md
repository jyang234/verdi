- **judged-ac3-kernel-violation-exit2** [UNDISPOSITIONED]: A manifest that strict-decodes cleanly but violates a kernel rule (unknown kind, absent obligations list, etc.) exits 2, grouped with 'undecodable' (cmd/verdi/model.go, TestModelCheck_KernelViolation_Exit2). ac-3 never enumerates this case; the implementation resolves the gap toward 'operational trouble', but under the module-wide 0/1/2 contract (verdict vs operational) and the spec's own framing that model check gives 'the same fail-closed feedback verdi lint already gives a spec' (lint verdicts exit 1), a kernel-invalid manifest is a verdict about artifact content, not operational trouble. The conflict with the plan's 'exit 1 on validation/frontier failure' is disclosed in comments, but the chosen reading stretches 'undecodable' to cover documents that did decode. (confidence 0.45)
- **judged-frontier-display-structural** [UNDISPOSITIONED]: checkFrontier treats Class.Display as structural: a display-label-only change (e.g. classes.feature.display: "Epic") trips the pinned frontier error (classesMatchFrontier compares Display; TestCheckFrontier_DisplayChangeTrips pins it). The spec defines the frontier as 'a different state set, transition set, class set, or obligation set' with vocabulary exempt as display-only — a changed display label is not a different class set, and Display is presentation-layer exactly like Vocabulary.Classes sitting above it. The implementation is stricter than the declared frontier on a field the spec never names; disclosed as a judgment call in model.go but invisible to a mechanical diff. (confidence 0.45)
- **judged-frontier-obligations-positional** [UNDISPOSITIONED]: transitionEqual compares obligations positionally, so a manifest listing the close transition's two obligations in reverse order trips the frontier error even though it describes the identical obligation set — while States/Terminal are deliberately compared as sets on the rationale that dc-1's text names 'sets, never order'. That same rationale applies to 'obligation set', so the positional posture is internally inconsistent with the spec's set language; disclosed in the code as stricter-than-required. (confidence 0.30)
- **judged-terminal-freeze-not-kernel** [UNDISPOSITIONED]: The outcome names 'terminal-states freeze' among the kernel validation rules, but no kernel rule forbids a transition whose `from` is a terminal state — Validate only checks terminal ⊆ states, and even counts terminal states as reachable by fiat. A model with a closed→draft transition is kernel-clean and is rejected today only by the frontier check; once dc-1's frontier lifts, nothing in the kernel preserves terminality. If 'freeze' merely abbreviates the subset rule this is moot, but read as 'terminal states admit no exits' the rule is unimplemented and untested. (confidence 0.35)
